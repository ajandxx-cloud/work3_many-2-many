可以。基于你开题报告中对 Work 3 的设定——“\*\*many-to-many 场景下，上车点/下车点动态分配与路径协同在线优化；小规模用精确算法验证，大规模用启发式算法保证速度\*\*”——我建议你把这一工作明确收敛为一篇\*\*以运营优化为主、以乘客响应为辨识度、以双侧接驳点为核心创新\*\*的论文，而不是把价格、菜单、强化学习全部一起塞进去。这样最符合你前两个工作的递进关系，也最容易形成一篇结构清楚、贡献突出的 TR Part C / TRB / TRE 风格论文。



我先说判断：\*\*你最合适的路线，不是做“普通 many-to-many DARP”，而是做“考虑乘客响应的 many-to-many 双侧接驳点分配与动态路径协同优化”\*\*。因为近年的前沿研究已经很明确地显示，meeting point 不应该只是外生给定，而应进入优化过程；同时，动态 DRT 的主流求解框架正在从单次插入，转向\*\*rolling horizon、多阶段重优化、时间依赖路网、以及兼顾用户体验的多目标框架\*\*。2024 年的 DARPmp 已把 meeting point selection 内生到模型中，并对大规模网络采用 Tabu/metaheuristics；2025 年的动态 DRT 研究则把车辆路径、乘客分配、发车时机放进 rolling horizon 统一优化；另有 2024 年工作把 hybrid requests、公平性、需求预测和 MARL 结合起来，说明“只看成本、不看乘客体验”的写法已经不够前沿了。(\[科学直通车]\[1])



但对你来说，\*\*不建议把强化学习设为 Work 3 的主方法\*\*。原因不是它不前沿，而是它会把论文重心带偏。你前两个工作已经形成了“乘客选择—菜单/接驳点—在线运营”的主线，Work 3 最需要的是把这条线从 many-to-one 扩展到 many-to-many，并把“上车点选择”自然升级为“\*\*上车点 + 下车点\*\*的双向分配”。如果此时再把 RL 当主角，论文会变成“算法论文”，而不是“问题创新 + 模型创新 + 算法适配”的运筹管理论文。更稳妥的做法是：\*\*主算法用 insertion + rolling horizon + ALNS/LNS，学习方法只作为加速模块或扩展讨论\*\*。这也更贴合你开题报告里的原始定位。 (\[IDEAS/RePEc]\[2])



我认为你这篇文章最值得写成下面这个题目方向：



\*\*中文方向\*\*：考虑乘客响应的多对多需求响应公交双侧接驳点分配与动态路径协同优化

\*\*英文方向\*\*：Online co-optimization of bidirectional meeting-point assignment and dynamic routing for many-to-many demand-responsive transit with passenger choice



这个题目好在三点。第一，它和 Work 1、Work 2 连得上：Work 1 是单侧上车点 + 定价，Work 2 是“接驳点+时间窗”菜单，Work 3 则自然升级为 many-to-many 下的\*\*双侧空间菜单\*\*。第二，它比传统 DARP 更有新意，因为你不是把乘客当被动需求，而是保留乘客对步行、等待、在途时间等属性的响应。第三，它比“定价 + 菜单 + 路径 + 预测 + RL 全包”更可落地。 (\[科学直通车]\[3])



具体建模上，我建议你采用一个\*\*中等复杂度、但论文辨识度很强\*\*的版本，而不是一步到位做最复杂版本。



你可以把每个请求 (r) 定义为：到达时刻、真实起点 (o\_r)、真实终点 (d\_r)、乘客类型 (g\_r)。系统不再只给一个 pickup 候选集，而是同时生成：



\* pickup 候选集 (M\_r^P)：原点周边可步行到达的上车接驳点；

\* dropoff 候选集 (M\_r^D)：终点周边可步行离开的下车接驳点。



然后，系统在线生成一组可行替代方案，例如

((m\_r^P, m\_r^D, \\hat t\_r^P, \\hat t\_r^D))，

也就是“上车点 + 下车点 + 预计接载时间/时间窗 + 预计送达时间”。

乘客在这些替代方案和 outside option 之间选择；平台则需要决定是否接受请求、给哪个车、插在哪两个位置、是否在当前 rolling horizon 内重优化全局路径。这个写法和你 Work 2 的“服务菜单”衔接得非常自然，只是从单侧菜单扩展成了\*\*双侧空间服务菜单\*\*。 (\[科学直通车]\[1])



这里有三个版本可以选，我最推荐第二个：



第一种是\*\*纯运营版\*\*：系统直接决定 (m\_r^P,m\_r^D,v\_r,\\pi\_r^P,\\pi\_r^D)，不显式建乘客选择。这个版本最容易做，但和你前两篇的主线联系不够强。



第二种是\*\*菜单—选择—插入版\*\*：系统先生成少量可行方案，乘客按 MNL 在这些方案和 outside option 中选择，平台再根据 realized choice 更新状态。这个版本最适合你，因为既保留“乘客响应”，又不会把模型复杂度推爆。



第三种是\*\*价格也内生的全统一版\*\*：把票价/补贴也放进 Work 3。这个版本理论上最完整，但我不建议你在主模型里做。因为价格已经是 Work 1、Work 2 的重点，再放到 Work 3 容易使文章过宽、贡献点发散。



所以我的建议很明确：\*\*Work 3 主文保留乘客选择，但先不把价格作为核心决策变量。\*\* 票价可以固定，或者只做“步行补偿折扣”的敏感性分析。这样你既保住了论文主线，也避免与 Work 1/2 重叠。



目标函数方面，你开题报告里已经写得很对了：

\[

\\min C^{op}+C^{wait}+C^{walk}+C^{IVT}+C^{rej}

]

我建议继续沿这个方向，但稍微再“论文化”一点：把它写成\*\*广义系统成本最小化\*\*或\*\*广义利润最大化\*\*。前者更稳，后者更容易和前两篇衔接。具体可以保留五类成本：



\* 车辆运行成本；

\* 等待成本；

\* 步行不便成本；

\* 车内时间/绕行成本；

\* 拒单惩罚。

&#x20; 如果你希望更前沿一些，可以加一个\*\*公平性项\*\*，例如最坏分位数等待时间、最大 detour ratio、或服务不均衡惩罚。因为近年的 exact DARP 文献已经把 equity-aware routing 提到了很重要的位置。 (\[IDEAS/RePEc]\[4])



在“前沿研究给你的方法启发”这一块，我认为最值得吸收的是五点。



第一，\*\*双侧接驳点必须内生\*\*。2024 年 DARPmp 的核心贡献，就是把 meeting point selection 真正纳入优化，而不是外生给定。你可以进一步把它从单侧扩展为 pickup/dropoff 双侧。(\[科学直通车]\[1])



第二，\*\*动态求解应以 rolling horizon 为骨架\*\*。2025 年动态 DRT 研究已经明确把 routing、assignment、departure timing 放进 rolling horizon 统一优化；2024 年共享接驳 taxi ridesharing 研究也采用 rolling horizon 来处理未来请求不确定性。你的 Work 3 若不用 rolling horizon，会显得偏旧。(\[IDEAS/RePEc]\[2])



第三，\*\*大规模算法不要只靠“简单贪婪插入”\*\*。更好的做法是“快速插入 + 局部修复 + horizon 内重优化”。一些动态 DARP 研究开始使用 multiple-plan 思想：不是始终维护一条当前最优路径，而是同时维护多个候选计划，从而提高后续请求插入成功率。这个思路很适合你在 many-to-many 下做 anticipatory enhancement。(\[Springer]\[5])



第四，\*\*用户体验指标正在变得更重要\*\*。近年的研究已不满足于最小化运营成本，开始显式建模 equity、perceived travel times、甚至拥挤导致的感知在途时间上升。对你来说，这意味着 Work 3 不能只汇报“成本下降多少”，还要汇报“步行、等待、绕行、公平性”是否改善。(\[IDEAS/RePEc]\[4])



第五，\*\*快速时序可行性检查是关键工程点\*\*。2024 年有工作提出了面向 DARP 的高效 scheduling heuristic，在最坏二次时间内优化 ride-time-oriented schedule，并强调它适合作为更大元启发式或精确算法的子模块。你完全可以把它当作“快速评估某条插入路线是否可行、其时序代价是多少”的底层算子。这样你的大算法会更像真正能跑起来的系统。(\[科学直通车]\[6])



基于这些前沿，我建议你的算法框架这样设计。



第一层是\*\*候选生成\*\*。

对每个新请求，生成有限个 pickup 候选点和 dropoff 候选点，而不是全网穷举。比如各取最近的 (k\_P,k\_D) 个可步行点，再做基础过滤：步行半径、道路可达性、最早最晚时刻、车容量粗筛。这样可以把原问题从“连续空间”压缩成“有限候选对”。这个思路和 meeting-point DARP、shared pickup/dropoff routing 的建模方向是一致的。(\[科学直通车]\[1])



第二层是\*\*在线插入评估\*\*。

对于每个候选 ((m^P,m^D))，枚举车辆 (v) 和插入位置 ((\\pi^P,\\pi^D))，用快速时序检查算法算出：



\* 是否可行；

\* 增量运营成本；

\* 预计等待时间；

\* 预计车内时间；

\* 预计绕行；

\* 可能影响到的已有乘客惩罚。

&#x20; 这里你可以继续沿用 Work 1 里“预测服务顺序/位置”的思想，但在 Work 3 中，它不再是只预测单点 marginal cost，而是要支持\*\*双点插入的边际代价评估\*\*。这是很自然的递进。(\[科学直通车]\[6])



第三层是\*\*乘客响应\*\*。

从所有可行备选里，保留前 (K) 个最优方案，形成实时服务菜单；乘客按效用函数选择。效用建议保留三项主属性：步行距离、等待时间、车内时间，再加 outside option。价格可以先固定。这样写的好处是：你的 Work 3 不会丢掉“用户选择”这条博士论文主线。并且从方法论上，它和 Work 2 的 assortment/menu 思路是连续的。



第四层是\*\*rolling horizon 重优化\*\*。

每隔 (\\Delta) 分钟，或每累计若干请求，就对未来窗口 (\[t,t+H]) 内的已接受但未完成服务做一次局部重优化。这里用 ALNS/LNS 最合适。destroy/repair 操作可以围绕：



\* 请求重分配；

\* pickup/dropoff 对替换；

\* 插入位置交换；

\* 路线片段重构；

\* 少量 rejected request reinsertion。

&#x20; 这会比“来一个单插一次”的方法明显更强，也更符合当前动态 DRT 的主流做法。(\[IDEAS/RePEc]\[2])



第五层是\*\*anticipatory enhancement\*\*。

这部分不必写得太重，但很值得作为你的“方法亮点”：可以基于历史到达率，采样若干未来虚拟请求场景，用它们评估当前插入方案的后悔值，或维护多个未来计划而非单一路径。这样你就不是纯 myopic online insertion，而是“light anticipatory rolling horizon”。这在论文层面会显著加分。(\[Springer]\[5])



小规模精确算法方面，我建议你不要追求“全动态精确求解”，而是做一个\*\*静态快照/批量窗口的精确 benchmark\*\*。也就是：在某个滚动窗口内，把已知请求当作静态实例，建立 MILP 或 set-partitioning / branch-cut-and-price 形式，求最优或近最优解，用来验证启发式的质量。近年的 exact DARP 文献表明，这条路完全可行，至少在几十个乘客规模上可以给出高质量 benchmark；比如 2025 年 equity-aware DARP 的 BCP 可以在 1 小时内最优求解大多数 55 名乘客、13 辆车、110 个节点的实例。你不一定要照搬它的公平性目标，但它说明“Work 3 做小规模 exact validation”是有研究基础的。(\[IDEAS/RePEc]\[4])



场景选择上，我建议你\*\*不要把 Work 3 做成典型 suburban feeder\*\*。因为最新参数化研究表明，suburban feeder 确实往往是 DRT 最具成本优势的应用场景，但它本质上更接近方向性很强的 many-to-one / one-to-many 结构。你的 Work 3 如果还选这种场景，会和前两篇过于接近。更好的选择是：



\* 小城市全域 many-to-many；

\* 大城市非高峰、低密度区域 many-to-many；

\* 园区/新区内部多点到多点接驳。

&#x20; 这样更能体现 many-to-many 的必要性，也更符合那篇 2026 研究对应用情境的划分。(\[科学直通车]\[7])



实验对比组我建议至少做四组。



第一组，\*\*Door-to-door DARP\*\*：不允许接驳点，作为传统基线。

第二组，\*\*单侧接驳点\*\*：只允许 pickup 灵活，dropoff 固定。

第三组，\*\*双侧接驳点但无乘客选择\*\*：系统直接指定点。

第四组，\*\*你的完整模型\*\*：双侧接驳点 + 乘客响应 + rolling horizon 协同优化。



若精力允许，再加两个消融：



\* 去掉 rolling horizon，只做 myopic insertion；

\* 去掉 anticipatory/multiple-plan，只保留单计划。



这样实验会非常有层次，也能把你的贡献拆得很清楚。(\[科学直通车]\[1])



评价指标不要只放运营成本。建议至少包括：



\* accepted requests / rejection rate；

\* vehicle-km / vehicle-hours；

\* average and 95th-percentile waiting time；

\* average walking distance at pickup/dropoff；

\* average in-vehicle time 和 detour ratio；

\* fairness 指标，如 worst-case detour 或 wait dispersion；

\* CPU time / average response time。

&#x20; 因为近年的 many-to-many / flexible bus / equity-aware 文献都在强调，\*\*成本、服务率、体验、公平性、实时性\*\*需要一起看。(\[科学直通车]\[8])



从论文贡献表述上，我觉得你最终可以凝练成这四条：



第一，提出一个\*\*考虑双侧接驳点与乘客响应的 many-to-many DRT 在线协同优化框架\*\*。

第二，把 Work 2 的“服务菜单”思想推广为 many-to-many 下的\*\*双侧空间服务菜单\*\*。

第三，设计“\*\*快速插入 + rolling horizon + ALNS + 轻量 anticipatory\*\*”的可扩展算法框架，并用小规模 exact benchmark 验证。
